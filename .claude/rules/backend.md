# Backend Rules — NestJS API (`apps/api`)

## Architecture

- **NestJS 11** with global module pattern. All modules registered in `app.module.ts`.
- **Drizzle ORM v0.39** with `postgres-js` driver. Schema lives in shared `packages/database`.
- **Database injection**: Use `@Inject(DB) private db: Database` (symbol from `database.module.ts`). Database is a `@Global()` module — no need to import `DatabaseModule` in feature modules.
- **Database type**: Import `Database` from `../database/database.types` (it's `ReturnType<typeof drizzle<typeof schema>>`).

## Module Structure

Every new feature follows this pattern:

```
src/{feature}/
  {feature}.module.ts      # NestJS module — imports nothing (DB is global)
  {feature}.controller.ts  # HTTP handlers only — no business logic
  {feature}.service.ts     # All business logic + DB queries here
```

Register every new module in `app.module.ts` imports array.

## Conventions

### Schema & Types
- Import schema tables from `@symph-crm/database` (e.g., `import { deals } from '@symph-crm/database'`).
- Use `typeof deals.$inferInsert` for create DTOs and `Partial<typeof deals.$inferInsert>` for update DTOs.
- All IDs are `uuid` (generated by Postgres via `defaultRandom()`). Never generate UUIDs in application code.
- All tables with user data MUST include `workspaceId` for multi-tenancy. Every query MUST filter by workspace.

### Controllers
- One controller per module, decorated with `@Controller('{resource}')`.
- Use standard REST verbs: `@Get()`, `@Post()`, `@Put(':id')`, `@Delete(':id')`.
- Controllers are thin — they call the service and return the result. No business logic.
- Use `@Param('id')` for path params, `@Body()` for request bodies, `@Query()` for filters.

### Services
- One service per module, decorated with `@Injectable()`.
- All DB queries use Drizzle's explicit query builder (`.select().from()`, `.insert().values()`, `.update().set()`). No magic — no `.findFirst()` unless using relational queries.
- Always return `.returning()` on insert/update for the created/updated record.
- Destructure single results: `const [result] = await this.db.select()...` (Drizzle returns arrays).
- Order queries with `orderBy(desc(table.createdAt))` by default for list endpoints.

### Error Handling
- Throw NestJS exceptions: `NotFoundException`, `BadRequestException`, `ForbiddenException`.
- Check existence before update/delete: if `findOne` returns undefined, throw `NotFoundException`.
- Never let raw Postgres errors bubble to the client.

### Environment
- Access env vars via `ConfigService` (injected via `ConfigModule.forRoot({ isGlobal: true })`).
- `.env` file path is `../../.env` (monorepo root). Never hardcode connection strings or secrets.
- Required env vars: `DATABASE_URL`. Optional: `ANTHROPIC_API_KEY`, `CRON_SECRET`.

### Naming
- Files: `kebab-case` (e.g., `deals.service.ts`, `pipeline-stages.controller.ts`).
- Classes: `PascalCase` with suffix (`DealsService`, `DealsController`, `DealsModule`).
- DB columns: `snake_case` in schema definitions, `camelCase` in TypeScript (Drizzle maps automatically).

## Database Migrations

- Generate migrations: `pnpm db:generate` (runs `drizzle-kit generate` from `packages/database`).
- Apply migrations: `pnpm db:migrate`.
- Schema changes go in `packages/database/src/schema/` — never write raw SQL migration files manually unless drizzle-kit is broken.
- Always add indexes for foreign key columns and frequently filtered columns.

## What NOT to Do

- Do NOT use Prisma, TypeORM, or raw SQL queries.
- Do NOT put business logic in controllers.
- Do NOT create API routes in the Next.js app (`apps/web/src/app/api/`). All API routes go through NestJS.
- Do NOT import from `apps/web` in the API. The only shared package is `@symph-crm/database`.
- Do NOT use `any` type. Use Drizzle's inferred types or define explicit interfaces.
- Do NOT commit `.env` files or credentials.
- Do NOT bypass workspace scoping — every data query must include workspace context.
